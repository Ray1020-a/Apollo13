# Agent 工作守則

> **每一輪 vibe coding，開工第一件事就是讀這份檔案。**
> 這是你的憲法。其他文件會變，這份很少變。

---

## 你是誰、你在做什麼

你在接力開發《阿波羅 13 號：極限斷電》。
你不是唯一一個 agent，前後輪可能是不同 session。
所以你**不能依賴記憶**，只能依賴這四份檔案。把它們當成你的外接大腦。

| 檔案 | 角色 | 你對它做什麼 |
|------|------|--------------|
| `AGENT_PROTOCOL.md`（本檔） | 憲法 | **讀**。除非守則本身要改，否則不動。 |
| `TASKS.md` | 看板，任務狀態唯一真相 | **讀**（撿任務）+ **寫**（移動狀態） |
| `PROGRESS_LOG.md` | 逐輪日誌，集體記憶 | **讀**（最近 1–2 筆）+ **附加**（每輪結尾） |
| `ARCHITECTURE.md` | 活的技術現況、系統合約 | **讀**（合約）+ **改**（結構變動時） |
| `SPEC_DECISIONS.md` | GDD 模糊處的工程裁定，全體遵守 | **讀**（動工前必查）+ **新增**（遇到新模糊時） |
| `../GDD.md` | 設計聖經，數值唯一來源 | **讀**。所有 magic number 來自這裡。 |

## 一輪的完整循環

### 階段一：認領（讀）

1. 讀本檔（`AGENT_PROTOCOL.md`）。
2. 讀 `TASKS.md`，找出 **BACKLOG 由上而下第一個「依賴全部在 DONE」的任務**。
   - 規則就這麼簡單：**檔案裡的順序 = 優先級**。撿最上面能做的那一個。
   - 不准跳到下面挑你喜歡的。不准同時開兩個任務。
3. 讀 `ARCHITECTURE.md` 拿到相關系統的合約。
4. 讀 `PROGRESS_LOG.md` 最近 1–2 筆，接上上一輪的上下文與坑。
5. 讀任務指向的 `GDD.md` 章節抄數值；再查 `SPEC_DECISIONS.md` 有沒有相關裁定 ——
   有就照辦，**別自己重新發明規則**。GDD 給數值，SPEC_DECISIONS 給「GDD 沒講清楚的部分怎麼算」。

### 階段二：執行（做）

6. 在 `TASKS.md` 把該任務從 BACKLOG 剪下、貼到 **IN PROGRESS**。一次只有一個。
7. 寫 code。對著任務的**驗收標準**做，不是對著感覺做。
8. 自我驗證（這是你能無人駕駛的關鍵）：
   ```
   npx tsc --noEmit      # 型別必須綠
   npx vitest run        # 系統測試必須綠
   ```
   - 系統類任務（P2）：**沒有對照 GDD 數值的測試 = 任務沒完成**。
   - **不要跑 `npm run dev` / vite dev server**，它不會結束，會把你卡死。
     視覺正確性留給人在瀏覽器肉眼確認。

### 階段三：回報（寫）

9. `TASKS.md`：把任務從 IN PROGRESS 移到 **DONE**，寫一行字 + 連到日誌那一筆。
10. `PROGRESS_LOG.md`：**附加**一筆（格式見該檔頂端），記：
    - 做了什麼、關鍵決定、踩到的坑、下一輪該注意什麼。
11. 結構/合約有變（新增檔案、改了 `GameState`、改了系統介面）→ 更新 `ARCHITECTURE.md`。
12. 過程中發現的新任務 → 加進 `TASKS.md` 的 BACKLOG，**放在它依賴解決後該在的位置**。

做完這 12 步，這一輪才算結束。

## 鐵律

1. **一次一個任務。** 撿 backlog 最上面能做的。不跳階、不囤積。
2. **數值只有一個來源：GDD。** 在 code 裡用具名常數，註解標出 GDD 章節，例如：
   ```ts
   // GDD 5-D: 無過濾器時 CO2 +20 PPM/秒
   const CO2_RISE_PER_SEC = 20;
   ```
3. **系統是純函式。** `game/systems/*` 裡不准出現 DOM、不准 `Math.random()` 以外的副作用。
   `(state, dt) => newState`。可測試是不可妥協的。
4. **邏輯與算繪解耦。** 換素材不該動到 `game/`。動到了就是架構錯了。
5. **M2 之前不碰素材。** 先好玩，再好看。GDD 很漂亮，但醜的草版要先能跑通整局。
6. **改壞了就回報。** 測試紅了、卡住了、發現 GDD 數值矛盾了 → 寫進 PROGRESS_LOG，
   別假裝沒事，別硬湊。下一輪的人要看得到。

## 任務粒度：別鍍金（altitude）

只做驗收標準寫的那件事，不多做。

- 一個系統大概 **50–150 行**。如果你發現它膨脹成 300 行、長出一堆抽象層、設定選項、
  「以後可能會用到」的擴充點 —— 停。那不是這個任務要的。
- **不做沒人要求的事**：不加 feature、不加設定檔、不過早抽象、不過早優化。
  這遊戲每幀就是幾十個浮點數加減，效能在被證明有問題前都是非問題。正確 + 清楚優先。
- 程式碼風格跟著周圍既有的走，別自創一套。
- 如果一個任務做到一半發現它其實是三個任務 —— 把它拆了，多出來的兩個寫進 `TASKS.md` BACKLOG，
  在 `PROGRESS_LOG` 記一筆「為何拆」，然後只完成原本最小的那塊。

## 卡關與例外處理

別硬湊、別假裝沒事。遇到狀況照下面走：

- **BACKLOG 最上面那個被依賴擋住** → 往下找第一個依賴已滿足的。若全被擋 → 停，在 `PROGRESS_LOG`
  寫清楚「全部 blocked，卡在哪個依賴」，回報給人。
- **GDD 數值自相矛盾，或沒講清楚** → 別自己默默猜。去 `SPEC_DECISIONS.md` 看有沒有裁定；
  沒有就**新增一條裁定**（標 🟡 暫定），寫下你的裁決與理由，照它做下去，並在回報時點名這條讓人複核。
- **測試過不了，因為 GDD 數字本身兜不攏** → 同上，寫進 SPEC_DECISIONS，不要為了讓測試變綠而偷改斷言遷就錯誤實作。
- **同一個問題試了兩次誠實的解法都不行** → 別再鬼打牆。停下，把兩次試了什麼、為何失敗寫進 `PROGRESS_LOG`，回報。

## 何時停下來問人 vs 自己決定

預設是**自己用合理預設往前做**，並記錄你的決定。只有這幾種才停下來等人：

1. 那是**設計取捨**而非工程正確性問題（會改變玩法手感，例如 SPEC_DECISIONS 裡標 🟡 的開放項要「定案」時）。
2. 動作**不可逆或對外**（刪檔、覆寫你沒建立的東西、推遠端）。
3. 兩次誠實嘗試後仍卡死。

反過來：型別怎麼命名、函式怎麼切、檔案放哪——這些**別問**，挑個合理的做，順手記下即可。

## Git 紀律（vibe coding 的安全網）

> 目前這個專案**還不是 git repo**。請人先 `git init`（或你取得同意後執行），再開始 T-001。
> 有版本控制，搞砸了才能一鍵回滾——這對 agent 接力開發是必需品，不是選配。

- **一個完成的任務 = 一個 commit**。訊息格式：`T-0xx: <任務標題>`。
- commit 訊息結尾加：`Co-Authored-By: Claude <noreply@anthropic.com>`
- **不要 push、不要開 PR**，除非人明講要。本地 commit 就好。

## 「完成」的定義（Definition of Done）

一個任務完成，當且僅當：

- [ ] 驗收標準每一條都成立
- [ ] `tsc --noEmit` 綠
- [ ] `vitest run` 綠（系統任務必須有新測試）
- [ ] `TASKS.md` 已移到 DONE
- [ ] `PROGRESS_LOG.md` 已附加當輪日誌
- [ ] 有結構變動的話 `ARCHITECTURE.md` 已更新
- [ ] 過程中做的新裁定已寫進 `SPEC_DECISIONS.md`
- [ ] 已 commit（`T-0xx: 標題`），沒有 push

少一項，任務就還沒完。
